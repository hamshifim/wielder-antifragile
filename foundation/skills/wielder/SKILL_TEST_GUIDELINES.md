# Testing Constraints & Live QA Protocols

- **The Live Execution Rule**: Tests assessing integration interfaces, Data Lake endpoints, or stochastic Large Language Models (LLMs) MUST inherently execute live data schemas. You MUST NOT use `unittest.mock` to artificially spoof remote model APIs or PySpark IO logic. If it interacts with reality, you verify reality.
- **Load-Test Supremacy**: The primary philosophy of Wielder's testing doctrine is that a true load test is fundamentally superior to isolated integration and unit tests. In fact, integration and unit tests are simply down-scaled modulations or subsets of a primary load test. Tearing down and reprovisioning a complete local environment qualifies directly as an integration test, just as deploying to an ephemeral cloud topology with restricted dataset sizes does. The infrastructure and validation rigor remain identically pure—only the scale and topological dimensions change.
- **Physical Sandboxing**: Live tests that write physical data arrays MUST securely map outputs strictly to ignored `/tmp/` artifacts or natively `.gitignore`'d Sandbox outputs (e.g. `explore_output/`) to explicitly prevent committing multi-megabyte artifact bloat into the git history.
- **Abstract Function Independence**: Tests MUST target pure leaf nodes iteratively. Assess module functions by feeding them strictly primitive independent datatypes (URI strings, integer dimensions, explicit configuration dictionaries) rather than demanding massive monolithic DAG Orchestration boots to execute a trivial test slice.
- **Analytical Pruning Parity**: When testing PySpark architectures, assertion testing MUST concretely validate that O(1) query-pruning logic functions correctly against `<global_conf_id>` dimension boundaries, proving the pipeline bypasses irrelevant namespaces.
- **Topological DAG Ordering & `ttX` Nomenclature**: When architecting Integration tests for complex pipeline DAGs, you are strongly encouraged to encapsulate tests inside strict Chronological Domain Directories (e.g., `tests/01_core/`, `tests/02_msa_search/`) to physically enforce architectural evaluation boundaries within PyTest. Additionally, developers are strongly encouraged to prefix test scripts natively with Topological Tiers (e.g., `test_tt1_orchestrator.py`, `test_tt2_msa.py`) reflecting their distinct functional depths. This guarantees PyTest organically executes foundational pipelines before touching downstream ML consumers. If the architecture evolves by inserting intermediate domains, developers should aggressively rename and re-number the downstream directories and files. Git tracking should not be a consideration; let Git do what it likes. You should not hesitate to renumber downstream architectures strictly to preserve raw `git log / git blame` history! Architectural accuracy, pure linear iteration, and the strict mathematical representation of the pipeline generally supersede historical file tracking!
- **The Multi-Stage Microservice Verification Sequence**: Before finalizing or merging architectural modifications against an orchestrated Wielder service or PyHocon configuration, developers MUST physically assert the entire `WieldService` orchestration pipeline across the underlying infrastructure layer. The universal QA loop entails:
    1. **Plan (`-w plan`)**: Assert the deployment graph PyHocon parsing sequence organically without mutation (Dry-Run). Fix any explicit syntax, fallback, or override issues.
    2. **Delete (`-w delete`)**: Aggressively purge existing namespaces, services, and workloads to enforce isolated evaluation states and test clean deletions.
    3. **Apply (`-w apply`)**: Stand up the live infrastructure completely on the chosen Surface (e.g. `docker`, `aws_eks`). Validate container startup dependencies and ConfigMap bindings.
    4. **Sanity Execution (`sanity.py & pytest`)**: Programmatically strike the established API endpoints (e.g. `gRPC localhost:50051`) with localized integration suites or isolated Python `sanity` probes to prove Data Lake IO and Network boundaries are functionally resolving.
